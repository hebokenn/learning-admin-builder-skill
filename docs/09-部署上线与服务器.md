# 09-部署上线和服务器：Vercel、Render、VPS 怎么选

部署就是把网站从你的电脑发布到别人能访问的地方。部署涉及账号、域名、费用和生产数据，人需要做最后授权；部署前检查、命令整理、环境变量清单和验证清单可以让 AI 先完成。

## 三种部署方式

| 方式 | 适合谁 | 优点 | 风险 |
|---|---|---|---|
| 本地预览 | 只给自己看 | 最简单 | 不能正式使用 |
| Vercel / Render | 小白和小团队 | 少管服务器，连接 GitHub 快 | 复杂后台任务受平台限制 |
| VPS 云服务器 | 有维护能力的团队 | 控制力强，能跑 PM2、Nginx、定时脚本 | 要自己管安全、备份、证书 |

## 推荐选择

第一版：

- Next.js 项目优先看 [Vercel](https://vercel.com/docs)。
- 需要长期后台服务、定时任务、数据库同平台管理，可看 [Render](https://render.com/docs)。
- 数据库优先用托管 PostgreSQL，例如 [Supabase](https://supabase.com/docs) 或 [Neon](https://neon.com/docs)。

给小团队的默认建议：

```text
先用 GitHub + Vercel/Render + 托管 PostgreSQL。
等后台任务、私有网络、服务器控制要求变强，再考虑 VPS。
```

正式生产且有维护人：

- Ubuntu VPS
- Node.js
- PostgreSQL
- PM2
- Nginx
- Certbot / Let's Encrypt
- 定时备份

## Vercel / Render 部署流程

1. 把代码上传到 GitHub。
2. 登录部署平台。
3. 选择 GitHub 仓库。
4. 填构建命令，例如 `npm run build`。
5. 填启动命令，具体看平台和项目。
6. 配置环境变量。
7. 点击部署。
8. 打开平台给的访问地址。
9. 检查登录、页面、API、导入。

给 AI 的提示词：

```text
我已经做好了培训进度跟进网站，现在想部署上线。
请读取我的 package.json 和项目结构，判断适合 Vercel、Render 还是 VPS。
请说明：
1. 为什么推荐这个平台
2. 部署前要检查什么
3. 构建命令是什么
4. 环境变量有哪些
5. 部署后要验证哪些页面
```

## 让 AI 做部署前准备

把这段给能读项目的 AI：

```text
请为这个培训进度后台准备部署方案，但不要直接操作生产环境。

请你完成：
1. 读取 package.json、框架、数据库和脚本
2. 判断 Vercel、Render、VPS 哪个更合适
3. 列出构建命令、启动命令、环境变量、数据库要求
4. 检查是否有 .env、数据库、真实表格或密钥误入仓库
5. 生成部署步骤和回滚步骤
6. 生成上线后验证清单
7. 明确哪些动作必须由人授权：创建账号、绑定域名、填生产密钥、正式发布
```

AI 应把结果写成 [部署前 Go/No-Go 检查表](../templates/部署前Go-No-Go检查表.md)。没有通过 Go/No-Go，不要正式上线。

## VPS 部署大纲

适合需要完整控制服务器且有运维能力的人。小白第一版不建议直接走 VPS；让 AI 先给出 Vercel / Render / 托管数据库方案。VPS 命令应作为 runbook 由 AI 或技术负责人执行。

### 1. 准备服务器

- 选择 Ubuntu LTS。
- 创建普通用户。
- 配置 SSH。
- 开启防火墙，只开放 SSH、HTTP、HTTPS。

参考：[Ubuntu Server](https://ubuntu.com/download/server)。

### 2. 安装 Node.js

参考：[Node.js 下载](https://nodejs.org/en/download)。

检查：

```bash
node -v
npm -v
```

### 3. 拉取代码并构建

```bash
git clone 【你的仓库地址】
cd 【项目目录】
npm install
npm run build
```

### 4. 用 PM2 管理进程

PM2 可以让 Node 网站在后台持续运行，并在崩溃后重启。参考：[PM2 Quick Start](https://pm2.keymetrics.io/docs/usage/quick-start/)。

常见命令：

```bash
pm2 start npm --name training-admin -- start
pm2 status
pm2 logs training-admin
pm2 restart training-admin
```

### 5. 用 Nginx 绑定访问入口

Nginx 负责把域名访问转发到 Node 服务。参考：[Nginx 文档](https://nginx.org/en/docs/)。

### 6. 配置 HTTPS

用 Certbot 申请证书。参考：[Certbot](https://certbot.eff.org/)。

任何登录、后台、表单页面都必须使用 HTTPS。

## 环境变量

常见环境变量：

```text
DATABASE_URL=数据库连接地址
AUTH_SECRET=登录加密密钥
EMAIL_API_KEY=邮件服务密钥
PUBLIC_SITE_URL=网站公网地址
```

不要上传 `.env` 到 GitHub。

## 备份

上线后必须有备份：

- 数据库每日备份。
- 上传文件备份。
- 备份存到服务器外部。
- 定期测试能否恢复。

最低要求：

```text
每日备份保留 7 天
每周备份保留 4 周
每月备份保留 6 个月
```

## 部署验收清单

- [ ] 网站首页可访问。
- [ ] 登录可用。
- [ ] API 返回正常。
- [ ] 数据库连接正常。
- [ ] 导入功能可用。
- [ ] 邮件测试通过。
- [ ] HTTPS 生效。
- [ ] 手机和电脑都能打开。
- [ ] 备份任务已配置。
- [ ] 管理员知道如何查看日志。

## Go/No-Go 判断

上线前必须能回答：

| 检查 | Go 条件 | No-Go 条件 |
|---|---|---|
| 构建 | 构建成功 | 构建失败 |
| 数据库 | 迁移可执行且有备份 | 无备份或迁移不确定 |
| 环境变量 | 生产变量齐全 | 缺关键变量或把密钥写进仓库 |
| 权限 | 管理员/普通用户隔离正确 | 无权限也能看敏感数据 |
| 导入 | 预览和错误行可用 | 直接写入且不能回滚 |
| 回滚 | 有旧版本或恢复方案 | 出错无法恢复 |
| 日志 | 能查看错误和导入记录 | 出错无处排查 |

如果出现 No-Go，先修复或降低范围，不要带病上线。
