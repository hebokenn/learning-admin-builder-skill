# 12-AI 如何使用这套方法论

这一章给 AI 看。完整教程适合人慢慢读，但 AI 更需要快速知道：先读什么、什么时候读什么、输出什么、哪些事情不能做。

## AI 的入口顺序

```text
AI_CONTEXT.md
  -> AI_MANIFEST.yaml
  -> 用户任务
  -> 目标项目文件
  -> 相关 docs / templates / references
```

不要一开始把所有长文档都读完。先用 `AI_CONTEXT.md` 建立工作框架，再按场景加载。

## AI 的工作状态

| 状态 | 触发 | 应该做什么 | 不应该做什么 |
|---|---|---|---|
| 需求不清 | 用户只说想做类似网站 | 问需求、整理范围 | 直接写代码 |
| 有资料 | 有代码、Excel、文档 | 先检查资料，再补问 | 重复问文件里已有答案 |
| 要方案 | 用户要可行性或计划 | 输出模块、页面、数据、权限、风险 | 只给泛泛建议 |
| 要实现 | 用户明确让你做 | 分阶段改代码和验证 | 一次性大改无验证 |
| 要部署 | 用户要上线 | 判断平台、环境变量、备份、验证 | 把密钥写进仓库 |
| 要复用 skill | 用户用 Codex / WorkBuddy / Kun | 指向 skill 或 portable prompt | 假设所有工具都支持同一种格式 |

## AI 每次开始前检查

在目标项目里工作时，先检查：

- 是否有 `AGENTS.md`、`README.md`、`docs/AI_HANDOFF.md`、`.workbuddy/skills/`。
- 是否有 `package.json`、框架版本、运行命令、测试命令。
- 是否有样例 Excel / CSV，且是否已经脱敏。
- 当前项目是否已有设计系统、UI 组件、权限方案、数据库方案。
- 当前仓库是否有未提交改动，不能误删用户改动。

## 需求澄清输出模板

当信息不足时，按这个格式输出：

```text
我先不写代码。为了把它做成可长期使用的后台管理网站，我需要确认这些：

1. 这个学习/培训项目的目标是什么？
2. 每天谁会登录后台？他们分别要做什么？
3. 学员数据来自哪里？有没有脱敏样例表？
4. 要跟踪哪些阶段、任务、成绩或证书？
5. 第一版必须上线的功能有哪些？哪些可以第二阶段做？
```

## 方案输出模板

当用户需要计划时，输出：

```text
## 项目理解
## 角色和权限
## 第一版模块
## 延后模块
## 页面清单
## 数据模型
## 导入和冲突规则
## API 和自动化脚本
## UI 方向
## 后台管理和维护
## 验收标准
## 风险和待确认问题
```

## 实现输出模板

当用户要求实现时，按小步执行：

1. 读取目标项目规则和现有结构。
2. 建立最小数据模型和样例数据。
3. 做首页、学员列表、学员详情、任务/进度页面。
4. 接入 API 和数据库。
5. 做导入预览、校验、日志。
6. 做角色权限和后台设置。
7. 加报表、通知、预约、评分等可选模块。
8. 本地运行、构建检查、浏览器检查。
9. 更新目标项目交接文件。

## AI 应该优先压缩的信息

AI 不需要背诵所有教程，但必须抓住这些结构：

```text
业务目标 -> 角色权限 -> 培训流程 -> 数据对象 -> 页面 -> API -> 脚本 -> 验收 -> 部署 -> 维护
```

如果上下文有限，优先保留：

- 需求和范围。
- 页面清单。
- 数据字段映射。
- 角色权限。
- 导入规则。
- 验收标准。

可以延后读取：

- 完整部署细节。
- 具体工具下载说明。
- 案例细节。
- 长提示词合集。

## 常见 AI 错误

| 错误 | 修正 |
|---|---|
| 一上来生成漂亮首页 | 改成后台工作台，直接显示指标、风险、待办 |
| 忽略 Excel 字段 | 先读取样例、做字段映射、确认唯一键 |
| 默认加预约和评分 | 先问是否是核心流程 |
| 所有人一个权限 | 设计角色、页面权限、API 权限 |
| 没有导入日志 | 加 import_jobs、import_errors、预览和确认 |
| 没有维护后台 | 加用户、角色、日志、备份、健康检查 |
| 没有移动端检查 | 检查 1280、1100、900、760、560 宽度 |
| 把私有信息写进公开仓库 | 只用假数据和方法论 |
