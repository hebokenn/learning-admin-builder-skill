# 06-前后端架构：代码应该怎么组织

这一章给 AI 和开发者看的重点是：代码怎么分层，才不会越写越乱。

## 推荐技术架构

推荐组合：

- 前端：Next.js + React + TypeScript
- 后端：Next.js Route Handlers
- 数据库：开发期 SQLite，正式多人使用建议 PostgreSQL
- UI：组件化后台界面
- 导入：Excel / CSV 预览和校验
- 通知：邮件或站内信
- 自动化：每日提醒、周报、备份、健康检查

简单理解：

```text
页面
  -> 调用 API
    -> 后端检查权限和参数
      -> 读写数据库
        -> 必要时触发通知、报表、脚本
```

Next.js 官方文档把 App Router、项目结构、Route Handlers、部署等放在核心文档中；做具体项目时应查看对应版本文档：[Next.js Docs](https://nextjs.org/docs)。

## AI 实施前项目盘点

在已有项目里动手前，AI 必须先完成项目盘点，不要直接新建一套结构。

AI 需要读取：

- 项目规则：`README`、`AGENTS`、handoff、memory、Cursor rules、WorkBuddy 规则。
- 技术栈：`package.json`、锁文件、框架版本、运行脚本。
- 目录结构：页面、API、组件、数据库、样式、测试。
- 现有能力：登录、权限、UI 组件、数据库、导入、报表。
- 当前状态：git 是否有未提交改动。

AI 输出：

```text
保留：
需要新增：
需要改动：
第一阶段文件清单：
风险：
验证命令：
```

可使用 [AI 执行前项目审查清单](../templates/AI执行前项目审查清单.md)。

## 每层负责什么

| 层 | 负责什么 | 不应该做什么 |
|---|---|---|
| 页面 | 展示数据、用户操作 | 不写复杂权限和导入规则 |
| 组件 | 表格、按钮、弹窗、筛选 | 不直接连数据库 |
| API | 接收请求、检查权限、返回结果 | 不写重复页面逻辑 |
| service | 业务规则，例如进度计算 | 不处理 UI |
| validator | 校验字段和导入行 | 不操作页面 |
| db | 连接数据库、执行查询 | 不包含页面样式 |
| scripts | 定时任务、备份、同步 | 不依赖浏览器 |

## 模块契约表

每个核心模块都要从页面贯穿到验证，避免只做了页面但没有数据，或只做了 API 但没人能用。

| 模块 | 页面 | API | service | validator | db | 权限 | 验证 |
|---|---|---|---|---|---|---|---|
| 首页看板 | 指标、风险、待办 | summary、risk | 统计和风险规则 | 筛选参数 | learners、progress、follow_ups | 按角色过滤 | 指标正确、无权限不泄露 |
| 学员列表 | 表格、筛选、批量操作 | learners list | 查询和分页 | 搜索、筛选 | learners、cohorts | 范围过滤 | 搜索、分页、移动端 |
| 学员详情 | 进度、备注、跟进 | learner detail、progress update | 进度计算 | 状态和备注 | learners、progress、notes | 自己/负责范围 | 更新记录和日志 |
| 导入 | 上传、映射、预览、确认 | import preview/commit | 解析、校验、冲突 | 行级校验 | import_jobs、import_errors | 管理员 | dry-run、错误行、日志 |
| 报表 | 周报、导出 | report weekly/export | 汇总和导出 | 周期和范围 | reports、progress | 导出权限 | 内容、下载、日志 |
| 后台设置 | 用户、角色、规则 | users、roles、settings | 管理配置 | 权限和字段 | users、roles、settings | 超管/管理员 | 无权限拦截 |

## 权限设计

建议先用这四类：

| 角色 | 权限 |
|---|---|
| admin | 所有项目、用户、导入、设置、报表 |
| trainer | 自己负责的班级、学员、任务、跟进 |
| learner | 只看自己的任务和反馈 |
| manager | 看团队汇总和风险，不改结构 |

让 AI 写代码时，要求它把权限放到统一函数里：

```text
canViewLearner(user, learner)
canEditProgress(user, progress)
canManageProgram(user, program)
canExportReport(user)
```

## 新员工培训架构示例

```text
HR 导入新员工
  -> learners / cohorts
分配培训计划
  -> programs / stages / tasks
记录完成情况
  -> progress
发现逾期
  -> follow_ups / notifications
经理看团队结果
  -> reports
```

## 3PP 培训架构示例

3PP 项目通常多一个外部组织和认证逻辑：

- organizations：合作方或外部公司
- certifications：认证结果、有效期、复训时间
- scorings：考试或演练评分
- notifications：证书到期提醒

如果只是一次性培训，可以先不用 `certifications`，用 `progress + reports` 就够。

## 给 AI 的架构提示词

```text
请根据我的培训项目需求，设计一个 Next.js 培训进度后台的代码架构。

请输出：
1. 推荐目录结构
2. 页面放在哪里
3. API 放在哪里
4. 数据库访问放在哪里
5. 权限判断放在哪里
6. 导入和校验逻辑放在哪里
7. 自动化脚本放在哪里
8. 第一版先做哪些文件

要求：
- 不要复制任何私有项目源码
- 用中文解释每个目录的作用
- 保持适合长期维护
```

## AI 必须输出的文件级计划

实施前让 AI 输出：

```text
第一阶段目标：
会新增的文件：
会修改的文件：
不会触碰的文件：
需要保留的现有模式：
验证命令：
如果失败如何回滚：
```

如果这个计划没有明确文件和验证，不要开始大范围改动。
