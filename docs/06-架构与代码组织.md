# 06-前后端架构：代码应该怎么组织

这一章给 AI 和开发者看的重点是：代码怎么分层，才不会越写越乱。

## 推荐技术架构

推荐组合：

- 前端：Next.js + React + TypeScript
- 后端：Next.js Route Handlers
- 数据库：开发期 SQLite，正式多人使用建议 PostgreSQL
- UI：组件化后台界面
- 导入：Excel / CSV 预览和校验
- 通知：邮件或站内信
- 自动化：每日提醒、周报、备份、健康检查

简单理解：

```text
页面
  -> 调用 API
    -> 后端检查权限和参数
      -> 读写数据库
        -> 必要时触发通知、报表、脚本
```

Next.js 官方文档把 App Router、项目结构、Route Handlers、部署等放在核心文档中；做具体项目时应查看对应版本文档：[Next.js Docs](https://nextjs.org/docs)。

## 每层负责什么

| 层 | 负责什么 | 不应该做什么 |
|---|---|---|
| 页面 | 展示数据、用户操作 | 不写复杂权限和导入规则 |
| 组件 | 表格、按钮、弹窗、筛选 | 不直接连数据库 |
| API | 接收请求、检查权限、返回结果 | 不写重复页面逻辑 |
| service | 业务规则，例如进度计算 | 不处理 UI |
| validator | 校验字段和导入行 | 不操作页面 |
| db | 连接数据库、执行查询 | 不包含页面样式 |
| scripts | 定时任务、备份、同步 | 不依赖浏览器 |

## 权限设计

建议先用这四类：

| 角色 | 权限 |
|---|---|
| admin | 所有项目、用户、导入、设置、报表 |
| trainer | 自己负责的班级、学员、任务、跟进 |
| learner | 只看自己的任务和反馈 |
| manager | 看团队汇总和风险，不改结构 |

让 AI 写代码时，要求它把权限放到统一函数里：

```text
canViewLearner(user, learner)
canEditProgress(user, progress)
canManageProgram(user, program)
canExportReport(user)
```

## 新员工培训架构示例

```text
HR 导入新员工
  -> learners / cohorts
分配培训计划
  -> programs / stages / tasks
记录完成情况
  -> progress
发现逾期
  -> follow_ups / notifications
经理看团队结果
  -> reports
```

## 3PP 培训架构示例

3PP 项目通常多一个外部组织和认证逻辑：

- organizations：合作方或外部公司
- certifications：认证结果、有效期、复训时间
- scorings：考试或演练评分
- notifications：证书到期提醒

如果只是一次性培训，可以先不用 `certifications`，用 `progress + reports` 就够。

```text
6. 导入和校验逻辑放在哪里

- 不要复制任何私有项目源码
- 用中文解释每个目录的作用

```
